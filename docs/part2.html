01

POM이 왜 필요한가

1편에서 만든 E2E 테스트 코드를 다시 보겠습니다. 처음에는 아래처럼 테스트 파일에 셀렉터를 직접 썼습니다.

POM 적용 전 — test_login_e2e.py
def test_login_success(page):
    page.goto("http://127.0.0.1:5000/login")
    page.fill("#username", "kim")      # 셀렉터 직접 작성
    page.fill("#password", "pass123")  # 셀렉터 직접 작성
    page.click("#login-btn")           # 셀렉터 직접 작성
    assert "dashboard" in page.url

def test_login_fail(page):
    page.goto("http://127.0.0.1:5000/login")
    page.fill("#username", "")           # 또 같은 셀렉터 반복
    page.fill("#password", "pass123")  # 또 같은 셀렉터 반복
    page.click("#login-btn")           # 또 같은 셀렉터 반복
    assert page.is_visible("#error")

테스트가 2개일 때는 문제가 없어 보입니다. 하지만 테스트가 10개, 20개로 늘어나면 어떻게 될까요?

⚠️
실제로 겪은 문제 개발 중에 HTML의 id="login-btn"을 id="submit-btn"으로 변경했습니다. 그러자 #login-btn을 쓰는 모든 테스트가 한꺼번에 실패했습니다. 테스트 파일 여러 곳을 일일이 찾아서 수정해야 했습니다. 테스트가 많아질수록 이 비용은 기하급수적으로 커집니다.

이 문제의 핵심은 "어디서 요소를 찾는가(셀렉터)"와 "무엇을 검증하는가(테스트 로직)"가 같은 파일에 섞여있다는 것입니다. POM 패턴은 이 두 가지를 명확히 분리합니다.

02

POM 패턴이란

Page Object Model은 웹 페이지의 요소(버튼, 입력창 등)와 동작(로그인, 로그아웃 등)을 클래스로 캡슐화하는 디자인 패턴입니다. 실무에서 GUI 자동화 테스트를 작성할 때 거의 표준처럼 사용됩니다.

📦
캡슐화
페이지의 셀렉터와 동작을 클래스 안에 숨깁니다. 테스트 코드는 내부 구현을 몰라도 됩니다.
🔧
단일 수정 포인트
HTML이 바뀌면 해당 page 파일 하나만 수정합니다. 테스트 파일은 건드릴 필요가 없습니다.
📖
가독성
테스트 코드가 시나리오처럼 읽힙니다. "무엇을 하는지"가 명확하게 보입니다.
♻️
재사용성
login() 메서드를 여러 테스트에서 공유합니다. 같은 코드를 반복 작성하지 않습니다.

핵심 아이디어

POM의 핵심은 간단합니다. "테스트 파일은 시나리오만, 페이지 파일은 구현만" 담당하도록 역할을 나누는 것입니다.

❌ POM 없이 — 역할이 섞임
# 테스트 파일에 셀렉터가 직접 박혀있음
def test_로그인(page):
  page.fill("#username", "kim")
  page.fill("#password", "pass123")
  page.click("#login-btn")
  assert "dashboard" in page.url

# HTML 바뀌면 여기도 수정해야 함
def test_실패(page):
  page.fill("#username", "")
  page.fill("#password", "pass123")
  page.click("#login-btn")  # 또 수정
  assert page.is_visible("#error")
✅ POM 적용 — 역할이 분리됨
# 테스트 파일은 시나리오만
def test_로그인(login_page):
  login_page.login("kim", "pass123")
  assert "dashboard" in login_page.get_current_url()

def test_실패(login_page):
  login_page.login("", "pass123")
  assert login_page.is_error_visible()

# HTML 바뀌면 login_page.py만 수정

프로젝트 구조 변화

POM 적용을 위해 tests/e2e/pages/ 폴더를 새로 추가했습니다. 기존 파일은 그대로 두고 구조만 확장했습니다.

tests/e2e/
  conftest.py    ← fixture 재설계 (login_page, logged_in 추가)
  test_login_e2e.py ← 테스트 코드 전면 개선 (셀렉터 제거)
  pages/           ← 새로 추가된 폴더
    __init__.py    ← pages/ 를 Python 패키지로 인식

    login_page.py  ← 로그인 페이지 셀렉터 + 동작
    dashboard_page.py ← 대시보드 페이지 셀렉터 + 동작
파일역할담당하는 것

dashboard_page.py 대시보드 페이지 객체 셀렉터 상수, get_welcome_text(), logout()

test_login_e2e.py 테스트 시나리오 무엇을 검증하는가만 담당, 셀렉터 없음

BasePage 구현

BasePage는 모든 페이지 클래스의 부모 클래스입니다. LoginPage와 DashboardPage가 공통으로 쓰는 기능을 여기에 모아두면, 각 페이지 클래스에서 중복 코드 없이 상속받아 쓸 수 있습니다.

BasePage
공통 기능: navigate(), get_current_url(), take_screenshot()
↙                        ↘
LoginPage
login(), is_error_visible()
DashboardPage
get_welcome_text(), logout()
tests/e2e/pages/base_page.py
class BasePage:
    """
    모든 페이지 클래스의 부모 클래스
    공통 기능을 여기에 모아두면 LoginPage, DashboardPage에서
    중복 코드 없이 상속받아 쓸 수 있습니다
    """

    def __init__(self, page):
        # page = Playwright 브라우저 탭 1개
        # self.page 로 저장해두면 모든 메서드에서 접근 가능
        self.page = page

    def navigate(self, url: str):
        # 브라우저를 해당 URL로 이동
        self.page.goto(url)

    def get_current_url(self) -> str:
        # 현재 브라우저 URL 반환
        # "dashboard" in get_current_url() 처럼 페이지 이동 검증에 사용
        return self.page.url

    def take_screenshot(self, name: str):
        # 수동으로 스크린샷을 찍고 싶을 때 사용
        self.page.screenshot(path=f"artifacts/screenshots/{name}.png")
💡
상속이란? 부모 클래스의 메서드를 자식 클래스가 그대로 물려받는 것입니다. class LoginPage(BasePage) 처럼 선언하면, LoginPage는 BasePage의 navigate(), get_current_url()을 추가 코드 없이 바로 쓸 수 있습니다.
05

LoginPage / DashboardPage 구현

LoginPage

로그인 페이지의 모든 셀렉터를 상수로 관리하고, 관련 동작을 메서드로 묶었습니다. 셀렉터를 상수로 쓰는 이유는 간단합니다. HTML이 바뀌면 상수값 하나만 수정하면 그 상수를 쓰는 모든 메서드가 자동으로 업데이트되기 때문입니다.

tests/e2e/pages/login_page.py
from .base_page import BasePage
# "." = 같은 폴더(pages/)를 의미합니다


class LoginPage(BasePage):
    # BasePage를 상속 → navigate(), get_current_url() 자동으로 사용 가능

    URL = "http://127.0.0.1:5000/login"

    # 셀렉터를 상수로 관리
    # HTML의 id 값과 정확히 일치해야 합니다 (하이픈/언더스코어 주의!)
    USERNAME_INPUT = "#username"   # id="username" 인 input 태그
    PASSWORD_INPUT = "#password"   # id="password" 인 input 태그
    LOGIN_BUTTON   = "#login-btn"  # id="login-btn" 인 button 태그
    ERROR_MESSAGE  = "#error"      # id="error" 인 에러 메시지 태그

    def navigate_to(self):
        # 로그인 페이지로 이동
        # BasePage의 navigate()를 호출합니다
        self.navigate(self.URL)

    def login(self, username: str, password: str):
        # 로그인 동작 전체를 하나의 메서드로 캡슐화
        # 테스트 코드에서는 login("kim", "pass123") 한 줄로 끝납니다
        self.page.fill(self.USERNAME_INPUT, username)
        self.page.fill(self.PASSWORD_INPUT, password)
        self.page.click(self.LOGIN_BUTTON)

    def get_error_message(self) -> str:
        # 에러 메시지 텍스트 반환
        # 예: "로그인 실패: 입력값을 확인하세요."
        return self.page.text_content(self.ERROR_MESSAGE)

    def is_error_visible(self) -> bool:
        # 에러 메시지가 화면에 보이는지 True/False 반환
        # is_visible()은 실제로 화면에 렌더링됐는지 확인합니다
        # (display:none 이면 False)
        return self.page.is_visible(self.ERROR_MESSAGE)

DashboardPage

tests/e2e/pages/dashboard_page.py
from .base_page import BasePage


class DashboardPage(BasePage):

    WELCOME_TEXT  = "#welcome"    # id="welcome" — "Welcome, kim" 텍스트
    LOGOUT_BUTTON = "#logout-btn" # id="logout-btn" — 로그아웃 버튼

    def get_welcome_text(self) -> str:
        # 환영 메시지 텍스트 반환
        # 예: "Welcome, kim"
        return self.page.text_content(self.WELCOME_TEXT)

    def logout(self):
        # 로그아웃 버튼 클릭
        # 클릭 후 server.py의 /logout 라우트가 로그인 페이지로 redirect합니다
        self.page.click(self.LOGOUT_BUTTON)
🔍
셀렉터 오타 주의 — 실제로 겪은 실수 구현 중 #login_btn (언더스코어)으로 잘못 입력해서 30초 타임아웃 에러를 겪었습니다. HTML의 id와 정확히 일치해야 합니다. 셀렉터를 상수로 관리하면 이런 오타를 한 곳에서만 잡으면 됩니다.
06

fixture 재설계

POM을 적용하면서 conftest.py도 함께 개선했습니다. 기존에는 page fixture만 썼다면, 이제는 page 위에 LoginPage 객체를 얹은 login_page fixture를 만들어서 테스트 준비를 더 자동화했습니다.

tests/e2e/conftest.py
import pytest
import sys, os

# pages/ 폴더를 import 경로에 추가
sys.path.insert(0, os.path.join(os.path.dirname(__file__)))

from pages.login_page import LoginPage
from pages.dashboard_page import DashboardPage


@pytest.fixture
def login_page(page):
    """
    로그인 페이지 객체를 준비해서 반환하는 fixture
    - page: pytest-playwright가 자동 제공하는 브라우저 탭
    - LoginPage 객체를 만들고 로그인 페이지로 자동 이동까지 수행
    - 테스트 함수에서 login_page 파라미터만 선언하면 자동 주입됩니다
    """
    lp = LoginPage(page)    # LoginPage 객체 생성
    lp.navigate_to()        # 로그인 페이지로 자동 이동
    return lp


@pytest.fixture
def logged_in(login_page):
    """
    이미 로그인된 상태의 DashboardPage를 반환하는 fixture
    - login_page fixture를 파라미터로 받아 재사용합니다 (fixture 중첩)
    - 로그아웃 테스트, 대시보드 검증처럼 로그인이 필요한 테스트에 사용
    """
    login_page.login("kim", "pass123")   # 로그인 수행
    return DashboardPage(login_page.page)  # 대시보드 페이지 객체 반환

fixture 중첩이 강력한 이유

logged_in fixture가 login_page fixture를 파라미터로 받는 것을 fixture 중첩이라고 합니다. pytest가 알아서 실행 순서를 정해줍니다.

테스트가 logged_in fixture를 요청하면 pytest가 자동으로:

1. page fixture 실행 → 브라우저 탭 생성
2. login_page fixture 실행 → LoginPage 객체 생성 + 페이지 이동
3. logged_in fixture 실행 → 로그인 수행 + DashboardPage 반환
4. 테스트 함수 실행
5. 브라우저 탭 자동 종료

테스트 코드 변화

POM과 fixture를 모두 적용한 후 테스트 코드가 어떻게 바뀌었는지 비교합니다. 가장 큰 변화는 테스트 파일에서 셀렉터가 완전히 사라졌다는 것입니다.

tests/e2e/test_login_e2e.py
import pytest


class TestLogin:
    # 클래스로 테스트를 묶으면 pytest 리포트에서 그룹으로 보입니다

    def test_정상_로그인(self, login_page):
        # login_page = conftest.py의 fixture가 자동 주입됩니다
        # 이미 로그인 페이지에 있는 상태로 시작합니다
        login_page.login("kim", "pass123")
        assert "dashboard" in login_page.get_current_url()

    def test_빈_아이디_에러(self, login_page):
        login_page.login("", "pass123")
        assert login_page.is_error_visible() is True

    def test_짧은_비밀번호_에러(self, login_page):
        login_page.login("kim", "123")
        assert login_page.is_error_visible() is True

    def test_아이디_비번_동일_에러(self, login_page):
        login_page.login("kim", "kim")
        assert login_page.is_error_visible() is True

    @pytest.mark.parametrize("username,password", [
        ("",    "pass123"),  # 빈 아이디
        ("kim", "123"),      # 짧은 비밀번호
        ("kim", "kim"),      # 아이디 = 비밀번호
        ("",    ""),         # 둘 다 빈값
    ])
    def test_로그인_실패_케이스들(self, login_page, username, password):
        login_page.login(username, password)
        assert login_page.is_error_visible() is True


class TestDashboard:
    # logged_in fixture = 이미 로그인된 상태로 시작

    def test_환영메세지_표시(self, logged_in):
        welcome = logged_in.get_welcome_text()
        assert "kim" in welcome

    def test_로그아웃_후_로그인페이지(self, logged_in):
        logged_in.logout()
        assert "login" in logged_in.get_current_url()
✅

회고

POM 패턴을 적용하면서 가장 크게 느낀 것은 "좋은 테스트 코드는 코드를 읽지 않아도 무슨 테스트인지 알 수 있어야 한다"는 점입니다. POM 적용 전 코드는 #username, #login-btn 같은 HTML 세부사항이 테스트 로직과 뒤섞여 있었습니다. 적용 후에는 login_page.login("kim", "pass123") 처럼 자연어에 가깝게 읽힙니다.

처음에는 "파일이 늘어나서 복잡해지는 것 아닌가?"라는 생각도 했습니다. 하지만 실제로 HTML 셀렉터가 바뀌는 상황을 겪으면서, 파일이 늘어나는 비용보다 유지보수 비용이 줄어드는 이점이 훨씬 크다는 것을 체감했습니다. 이것이 POM 패턴이 실무에서 표준으로 자리잡은 이유라고 생각합니다.
