01

CI/CD가 뭔가

CI(Continuous Integration, 지속적 통합)는 코드를 변경할 때마다 자동으로 빌드하고 테스트를 실행하는 것입니다. CD는 문맥에 따라 Continuous Delivery(배포 가능한 상태까지 자동화) 또는 Continuous Deployment(실제 운영 배포까지 자동화)로 나뉩니다.

이 글에서는 CI(자동 테스트 실행)까지만 다룹니다.

🔄
CI (지속적 통합)
코드 변경 → 자동 테스트 실행 → 결과 알림. 문제를 빠르게 발견합니다.
🚀
CD (지속적 배포)
테스트 통과 → 자동 배포. 사람이 직접 배포 명령을 내리지 않아도 됩니다.
🛡️
이 프로젝트 범위
CI에 집중. push하면 pytest가 자동으로 18개 테스트를 실행합니다.
02

GitHub Actions가 왜 필요한가

1~2편에서 테스트를 완성했습니다. 그런데 테스트를 수동으로 돌리는 구조에선, 일정이 바쁘면 실행이 누락되기 쉽습니다. 그래서 자동 실행까지 연결해야 테스트가 기능합니다.

❌ GitHub Actions 없을 때
# 코드 수정할 때마다 사람이 직접
1. 터미널 열기
2. Flask 서버 켜기
3. pytest tests/ -v 입력
4. 결과 확인

# 바쁘면 테스트 실행을 잊음
# 팀원이 push해도 아무도 모름
# 테스트가 있으나 마나
✅ GitHub Actions 있을 때
# 코드 수정 후
git push

# GitHub이 자동으로:
# 1. 가상 서버 생성
# 2. 패키지 설치
# 3. Flask 서버 실행
# 4. pytest 18개 테스트 실행
# 5. 통과 ✅ / 실패 ❌ 표시
# → 사람이 아무것도 안 해도 됨
💡
테스트를 CI에 연결하면 코드 변경 시 자동으로 실행됩니다. 아무리 좋은 테스트 코드를 만들어도 실행을 강제하는 시스템이 없으면 흐지부지됩니다. GitHub Actions를 사용하면 코드가 push될 때마다 테스트가 자동으로 실행됩니다.
03

전체 흐름

git push 이후 GitHub Actions가 어떤 순서로 동작하는지 전체 흐름입니다.

💻
git push
로컬에서 코드 올림
→
🖥️
가상 서버 생성
GitHub ubuntu 서버
→
📦
환경 설치
Python + 패키지 + Playwright
→
🌐
서버 실행
Flask 백그라운드 실행
→
🧪
테스트 실행
18개 자동 실행
→
✅
결과 표시
통과 / 실패

이 전체 과정이 push 한 번으로 자동 실행됩니다. 실행 시간은 환경과 테스트 규모에 따라 달라집니다. (레포 기준 실제 평균 시간을 측정해 두면 더 좋습니다.)

04

workflow 파일 작성

GitHub Actions는 .github/workflows/ 폴더 안의 .yml 파일을 읽어서 실행합니다. workflow 파일에서는 테스트를 언제 실행할지와 어떤 작업을 수행할지 설정합니다.

.github/workflows/test.yml
# workflow 이름 — GitHub Actions 탭에 표시됩니다
name: Run Tests

# 언제 실행할지 정의합니다
on:
  push:
    branches: [ master ]    # master 브랜치에 push할 때
  pull_request:
    branches: [ master ]    # master으로 PR을 올릴 때

jobs:
  test:
    runs-on: ubuntu-latest  # GitHub이 제공하는 가상 서버 OS

    steps:

      # Step 1. 코드 가져오기
      # GitHub 서버에는 우리 코드가 없습니다.
      # checkout이 레포지토리를 서버로 복사해줍니다.
      - name: 코드 체크아웃
        uses: actions/checkout@v3

      # Step 2. Python 설치
      # 가상 서버에 Python이 없으므로 직접 설치합니다.
      - name: Python 설치
        uses: actions/setup-python@v4
        with:
          python-version: '3.11'

      # Step 3. 패키지 설치
      # requirements.txt에 있는 flask, pytest, playwright 등을 설치합니다.
      - name: 패키지 설치
        run: pip install -r requirements.txt

      # Step 4. Playwright 브라우저 설치
      # Playwright는 패키지 설치 외에 브라우저 바이너리도 별도로 설치해야 합니다.
      # --with-deps = 리눅스 환경에 필요한 시스템 의존성도 함께 설치
      # chromium만 설치해서 불필요한 firefox, webkit 설치를 생략합니다.
      - name: Playwright 브라우저 설치
        run: python -m playwright install --with-deps chromium

      # Step 5. Flask 서버 백그라운드 실행
      # & = 백그라운드 실행 (서버를 켜놓고 다음 단계로 넘어감)
      # sleep 2 = 서버가 완전히 뜰 때까지 2초 대기
      - name: Flask 서버 실행
        run: |
          cd app
          python server.py &
          sleep 2

      # Step 6. 단위 테스트 실행
      - name: 단위 테스트
        run: pytest tests/unit/ -v

      # Step 7. E2E 테스트 실행
      # 서버에는 화면이 없으므로 headless 모드로 실행됩니다. (기본값)
      - name: E2E 테스트
        run: pytest tests/e2e/ -v

      # Step 8. 실패 시 스크린샷 업로드
      # if: failure() = 앞 단계 중 하나라도 실패했을 때만 실행
      # artifacts = GitHub Actions가 파일을 보관해주는 저장소
      # 실패 원인 파악을 위해 스크린샷을 자동으로 수집합니다.
      - name: 스크린샷 업로드 (실패 시)
        if: failure()
        uses: actions/upload-artifact@v3
        with:
          name: screenshots
          path: artifacts/screenshots/

각 step이 하는 일 한눈에 보기

1
코드 체크아웃
GitHub 서버에는 우리 코드가 없습니다. actions/checkout이 레포지토리를 가상 서버로 복사해줍니다.
2
Python 설치
가상 서버는 매번 새로 만들어지는 깨끗한 환경입니다. Python도 직접 설치해야 합니다. 버전을 고정하면 "내 PC에선 되는데 서버에선 안 돼요" 문제를 방지합니다.
3
패키지 설치
requirements.txt에 정의된 flask, pytest, playwright 등을 설치합니다. requirements.txt가 있어야 하는 이유가 바로 여기 있습니다.
4
Playwright 브라우저 설치
Playwright는 pip 설치 외에 브라우저 바이너리도 별도 설치가 필요합니다. --with-deps는 Linux 환경에서 필요한 시스템 라이브러리도 함께 설치합니다.
5
Flask 서버 실행
&로 백그라운드 실행한 뒤 sleep 2로 서버가 뜰 때까지 대기합니다. 이 프로젝트는 서버 기동이 빨라 sleep 2로도 문제없이 동작했습니다. 다만 서버 초기화가 길어지는 환경에서는 테스트가 불안정해질 수 있어, “서버 응답 확인 후 테스트 시작” 방식으로 개선하는 편이 안정적입니다.
6~7
테스트 실행
단위 테스트(8개)와 E2E 테스트(10개)를 순서대로 실행합니다. CI 러너는 디스플레이가 없는 환경이라, 브라우저는 headless로 실행되는 구성이 일반적입니다. (프로젝트 설정에 따라 headed도 가능하지만, 기본은 headless로 두는 편이 안정적입니다.)
8
실패 시 스크린샷 업로드
테스트가 실패했을 때만 스크린샷을 GitHub artifacts에 업로드합니다. 1~2편에서 만든 스크린샷 자동 저장 기능이 여기서 빛을 발합니다.
05

서버 대기 전략 — sleep 2의 한계

sleep 2는 단순히 2초를 기다리는 방식입니다. 이 프로젝트의 Flask 앱은 가볍기 때문에 이 방식으로도 문제없이 동작했습니다. 다만 서버가 준비되기 전에 테스트가 시작되면 실패할 수 있기 때문에 서버가 실제로 응답하는지 확인한 뒤 테스트를 시작하는 방식도 사용할 수 있습니다.

💼 다른 방법

sleep으로 일정 시간을 기다리는 방식은 간단하지만 서버 준비 시간이 예상보다 길어질 경우 테스트가 먼저 실행될 수도 있습니다. 이럴 때 sleep 2는 서버가 아직 안 떴는데 테스트를 실행해서 실패하는 원인이 됩니다.

이럴 때는 서버가 실제로 응답할 때까지 확인한 뒤 다음 단계로 넘어가는 방식도 사용할 수 있습니다.

서버 응답 확인 후 테스트 시작
# sleep 2 대신 — 서버가 실제로 응답할 때까지 기다림
- name: Flask 서버 실행
  run: |
    cd app
    python server.py &
    python -c "
    import urllib.request, time
    for i in range(10):           # 최대 10초 시도
        try:
            urllib.request.urlopen('http://127.0.0.1:5000/login')
            print('서버 준비 완료')
            break                     # 응답 오면 즉시 다음 단계로
        except:
            print(f'대기 중... {i+1}초')
            time.sleep(1)             # 1초 후 재시도
    "

# 장점:
# - 서버가 뜨는 즉시 다음 단계로 넘어감 (불필요한 대기 없음)
# - 서버가 늦게 떠도 최대 10초까지 기다려줌 (안정성)
# - Python 내장 라이브러리만 사용 (별도 설치 불필요)
🔍
이 프로젝트에서 sleep 2를 유지한 이유 Flask 서버가 빠르게 실행되기 때문에 단순히 2초를 기다리는 방식으로도 충분했습니다. 다만 서버 준비 시간을 확인해야 하는 상황에서는 위와 같은 방식으로 변경할 수 있습니다.
06

실패 시 스크린샷 수집

2편에서 만든 pytest hook이 CI 환경에서 그대로 동작합니다. 테스트가 실패하면 artifacts/screenshots/에 스크린샷이 저장되고, GitHub Actions가 이를 artifacts로 업로드합니다.

테스트 실패 → 스크린샷 → artifacts 업로드 흐름
# 1. E2E 테스트 실패
# 2. tests/conftest.py의 hook이 자동으로 스크린샷 저장
#    → artifacts/screenshots/test_정상_로그인[chromium].png

# 3. workflow의 마지막 step이 실행됨
- name: 스크린샷 업로드 (실패 시)
  if: failure()     # 실패했을 때만
  uses: actions/upload-artifact@v3
  with:
    name: screenshots
    path: artifacts/screenshots/

# 4. GitHub Actions 탭 → 해당 workflow 클릭
#    → Artifacts 섹션에서 screenshots.zip 다운로드
#    → 실패 화면 즉시 확인 가능
📸
왜 스크린샷을 저장했는가 CI 환경에서는 브라우저가 화면 없이 실행되기 때문에 테스트가 실패해도 실제 화면을 바로 확인하기 어렵습니다. 스크린샷을 함께 저장해 두면 어떤 화면에서 실패했는지 바로 확인할 수 있습니다.

08

이 구조를 사용하면

이 구조를 사용하면 테스트를 자동으로 실행하고 결과를 확인하는 과정을 한 번에 관리할 수 있습니다.

🖥️
제품 GUI 검증 자동화에서 CI가 중요한 이유 GUI 기능은 코드 변경 한 줄로 의도치 않게 깨질 수 있습니다. 특히 여러 화면이 연결된 제품에서는 로그인 버튼 하나의 id가 바뀌었을 때 연관된 모든 시나리오가 동시에 실패할 수 있습니다. CI 파이프라인은 이런 회귀(regression)를 push 시점에 즉시 잡아냅니다.
🔒
브랜치 보호 규칙
GitHub 설정에서 "테스트 통과해야만 master 브랜치에 merge 가능"하도록 설정할 수도 있습니다.
📸
GUI 회귀 감지
제품 UI 변경 시 영향을 받는 테스트가 자동으로 실패하고 스크린샷이 남습니다. 어느 화면이 어떻게 깨졌는지 즉시 확인할 수 있습니다.
📊
테스트 리포트
Allure, pytest-html 등과 연동해서 테스트 결과를 시각적인 HTML 리포트로 자동 생성합니다.
🔔
Slack 알림
테스트 실패 시 Slack으로 알림을 보내도록 설정합니다. 팀원 모두가 즉시 인지할 수 있습니다.

이번 시리즈에서는 테스트 코드를 작성하는 것에서 시작해 브라우저 자동화, 그리고 GitHub Actions를 이용한 CI 연결까지 전체 흐름을 직접 구현해 보았습니다. 단순히 테스트 코드를 작성하는 것에서 끝나는 것이 아니라, 코드가 push될 때마다 테스트가 자동으로 실행되는 구조까지 만들어 보는 것이 목표였습니다.

특히 테스트를 자동으로 실행하는 환경을 만들어 보면서, 테스트가 코드와 함께 계속 실행되는 것이 중요하다는 것을 느꼈습니다. 로컬에서만 테스트를 실행하는 경우에는 상황에 따라 실행을 놓치기 쉽지만, CI에 연결해 두면 코드 변경이 생길 때마다 동일한 방식으로 테스트가 반복됩니다.

또한 서버가 준비되기 전에 테스트가 실행될 수 있다는 점이나, 실패했을 때 화면을 확인하기 어려운 문제 등을 직접 경험하면서 테스트 환경을 조금씩 보완해 나갈 수 있었습니다. 이번 과정은 테스트 코드를 작성하는 것뿐 아니라, 테스트가 실제로 돌아가는 환경을 함께 구성해 보는 경험이었습니다.
