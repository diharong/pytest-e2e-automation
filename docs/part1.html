01

왜 이 프로젝트를 만들었나

처음에는 "테스트 코드를 작성할 줄 안다"는 것을 보여주기 위해 시작했습니다. 하지만 단순히 이미 만들어진 서비스에 테스트를 붙이는 것은 "이 사람이 테스트를 이해하고 있는가"를 증명하기에 부족하다고 생각했습니다.

그래서 방향을 바꿨습니다. 테스트 대상 서비스(Flask 앱)부터 테스트 프레임워크(pytest + Playwright)까지 직접 설계하고 연결하는 것이 실무에서 QA 엔지니어나 개발자가 실제로 하는 일이기 때문입니다.

💡
테스트 자동화가 왜 필요한가 서비스 규모가 커질수록 수동 테스트는 한계에 부딪힙니다. 로그인 기능 하나를 수정했는데 회원가입이 깨지는 "회귀 버그"를 매번 사람이 직접 클릭해서 확인하는 건 비효율적입니다. 테스트 자동화는 이 반복 검증을 코드로 대체해서, 개발자가 변경을 두려워하지 않고 빠르게 배포할 수 있는 안전망이 됩니다.

이 프로젝트에서는 로그인/로그아웃이 있는 Flask 웹앱을 만들고, 로그인 성공/실패/엣지케이스를 자동으로 검증하는 테스트 스위트를 구축했습니다.

02

왜 Playwright인가

GUI 자동화 프레임워크는 Selenium, Cypress, Playwright 등 여러 선택지가 있습니다. 단순히 "많이 쓰니까"가 아니라, 각 도구의 특성을 비교하고 이 프로젝트에 가장 적합한 것을 선택했습니다.

프레임워크언어속도pytest 연동특징
Selenium Python, Java 등 보통 가능, 설정 복잡 가장 오래됨. 레거시 환경에 많이 쓰임
Cypress JavaScript 전용 빠름 불가 (JS 전용) Python 프로젝트와 연동 불가
Playwright ✓ Python, JS 등 빠름 네이티브 지원 Microsoft 개발. 빠르게 업계 표준이 되는 중

선택한 3가지 이유

1. pytest와 네이티브 통합

pytest-playwright를 설치하면 page, browser fixture가 자동으로 제공됩니다. Selenium은 WebDriver를 직접 생성하고 종료해야 하지만, Playwright는 pytest의 fixture 시스템에 자연스럽게 녹아드는 구조라서 테스트 코드가 훨씬 간결합니다.

Selenium vs Playwright 비교
# ❌ Selenium — 브라우저 설정을 직접 해야 함
from selenium import webdriver
driver = webdriver.Chrome()           # 드라이버 직접 생성
driver.get("http://localhost:5000/login")
driver.find_element(By.ID, "username").send_keys("kim")
driver.find_element(By.ID, "login-btn").click()
driver.quit()                         # 직접 종료 필요

# ✅ Playwright — fixture가 브라우저를 자동 관리
def test_login(page):              # page fixture 자동 주입
    page.goto("http://localhost:5000/login")
    page.fill("#username", "kim")
    page.click("#login-btn")
    # 브라우저 종료도 자동 처리됨

2. 자동 대기 (Auto-wait)

Selenium에서 흔히 발생하는 문제 중 하나가 "요소가 아직 로드되지 않았는데 클릭을 시도해서 실패"하는 타이밍 이슈입니다. 이를 해결하려면 time.sleep()이나 WebDriverWait 같은 대기 코드를 수동으로 추가해야 합니다. Playwright는 요소가 화면에 나타날 때까지 자동으로 기다려주므로, 별도의 대기 코드 없이 안정적인 테스트가 가능합니다.

3. Python 생태계와의 궁합

이 프로젝트는 Python(Flask)으로 만든 앱을 Python(pytest)으로 테스트하는 구조입니다. Cypress는 JavaScript 전용이라 선택지에서 제외됩니다. Selenium은 Python을 지원하지만 pytest와의 통합이 번거롭습니다. 같은 언어, 자연스러운 통합, 최신 기술 스택이라는 점에서 Playwright가 최적이었습니다.

🔍
업계 트렌드 2023~2024년 기준으로 Playwright는 Selenium을 빠르게 대체하고 있습니다. Microsoft가 적극적으로 유지보수하고 있고, Chromium/Firefox/WebKit 모든 브라우저를 단일 API로 지원한다는 점도 큰 장점입니다.
03

프로젝트 구조

테스트를 unit(단위 테스트)과 e2e(End-to-End 테스트)로 분리한 것이 이 프로젝트의 핵심 설계입니다. 두 가지를 같은 파일에 섞으면 관심사가 뒤섞여 유지보수가 어려워집니다.

pytest-gui-automation/
  app/
    __init__.py
    auth.py       ← 로그인 검증 로직 (UI와 분리)
    server.py     ← Flask 웹 서버 및 라우팅
    templates/
      login.html   ← id 속성으로 셀렉터 설계
      dashboard.html
  tests/
    conftest.py   ← 실패 시 스크린샷 자동 저장 (hook)
    unit/
      test_auth.py ← 단위 테스트 (브라우저 없음, 빠름)
    e2e/
      test_login_e2e.py ← GUI 자동화 테스트
  pytest.ini     ← 테스트 설정 및 marker 등록

⚡
실행 속도 차이
단위 테스트는 0.1초, E2E는 5~10초. 개발 중엔 단위 테스트만 빠르게 돌리고, 배포 전에만 E2E까지 실행합니다.
🎯
실패 원인 파악
단위 테스트 실패 = 로직 오류, E2E 실패 = UI/연동 문제. 분리되어 있으면 원인을 빠르게 좁힐 수 있습니다.
🔧
독립적 유지보수
HTML 구조가 바뀌어도 단위 테스트는 영향받지 않습니다. 각 테스트는 자신의 관심사에만 집중합니다.

pytest.ini 설정

pytest.ini를 두면 실행 옵션을 매번 입력하지 않아도 됩니다. markers 등록은 테스트를 목적에 따라 선택적으로 실행할 수 있게 해줍니다.

pytest.ini
[pytest]
testpaths = tests      # tests/ 폴더만 탐색
pythonpath = .         # 루트를 Python 경로에 추가 (import 오류 방지)
markers =
    smoke: 핵심 기능만 빠르게 검증
    regression: 전체 회귀 테스트
    e2e: 브라우저 기반 GUI 자동화
💡
marker를 실무에서 사용하는 방법
배포 전 빠른 확인: pytest -m smoke (수십 초)
전체 검증: pytest -m regression (수 분)
GUI만 실행: pytest -m e2e --headed
04

테스트 대상 앱 만들기

GUI 자동화는 테스트할 화면이 먼저 있어야 합니다. 단순히 동작하는 앱이 아니라, 테스트하기 좋은 구조로 설계하는 것이 핵심입니다.

설계 포인트 1: 로직과 UI의 분리

로그인 검증 로직을 auth.py에 분리했습니다. Flask(server.py)는 HTTP 요청을 받아 auth.py의 함수를 호출하는 역할만 합니다. 이렇게 하면 단위 테스트에서 브라우저 없이 함수만 직접 호출해서 빠르게 검증할 수 있습니다.

app/auth.py
def validate_login(username: str, password: str) -> bool:
    """
    로그인 검증 로직 — Flask와 완전히 분리
    단위 테스트에서 브라우저 없이 직접 호출 가능
    """
    if not username:           # 빈 아이디 거부
        return False
    if len(password) < 4:     # 4자 미만 비밀번호 거부
        return False
    if username == password:   # 아이디와 비밀번호 동일 거부
        return False
    return True

설계 포인트 2: HTML에 id 속성 부여

Playwright가 화면 요소를 찾으려면 고유한 식별자가 필요합니다. CSS 클래스(.btn)는 여러 요소에 중복될 수 있어 불안정합니다. id="login-btn"처럼 id를 처음부터 설계해두면 #login-btn으로 안정적으로 선택할 수 있습니다.

app/templates/login.html (핵심 부분)
<form method="post" action="/login">
  <!-- id 속성이 있어야 Playwright가 안정적으로 찾을 수 있음 -->
  <input id="username" name="username" />
  <input id="password" name="password" type="password" />
  <button id="login-btn" type="submit">Login</button>
</form>

<!-- 에러 메시지도 id 부여 → 테스트에서 is_visible()로 감지 가능 -->
{% if error %}
  <p id="error">{{ error }}</p>
{% endif %}
⚠️
테스트를 고려한 UI 설계 실무에서도 개발자와 QA가 협업할 때 "이 버튼에 id를 붙여주세요"라는 요청이 자주 나옵니다. 처음부터 테스트 가능한 구조로 만드는 습관이 중요합니다.
05

단위 테스트 (pytest)

E2E 테스트를 짜기 전에, 비즈니스 로직 자체가 올바른지 먼저 단위 테스트로 검증했습니다. 브라우저 없이 실행되므로 매우 빠르고, 로직 오류를 초기에 잡아낼 수 있습니다.

assert — 기댓값 검증의 기본

pytest에서 assert는 테스트의 핵심입니다. 조건이 False이면 테스트를 실패시키고, 어떤 값이 기대와 달랐는지 상세히 출력해줍니다.

tests/unit/test_auth.py
import pytest
from app.auth import validate_login

# 성공 케이스
def test_정상_로그인_성공():
    assert validate_login("kim", "pass123") is True

# 실패 케이스 — 경계값을 하나씩 검증
def test_빈_아이디_실패():
    assert validate_login("", "pass123") is False

def test_짧은_비밀번호_실패():
    assert validate_login("kim", "123") is False   # 3자: 실패

def test_아이디_비밀번호_동일_실패():
    assert validate_login("kim", "kim") is False

parametrize — 반복 케이스를 데이터로 분리

같은 로직을 다양한 입력값으로 검증할 때, 테스트 함수를 여러 개 만들면 중복이 발생합니다. @pytest.mark.parametrize를 사용하면 입력 데이터만 나열하고 테스트 로직은 하나로 유지할 수 있습니다. 실무에서 경계값 테스트 작성 시 반드시 쓰는 패턴입니다.

tests/unit/test_auth.py
@pytest.mark.parametrize("username, password, expected", [
    ("kim",  "pass123", True),   # 정상 케이스
    ("",    "pass123", False),  # 빈 아이디
    ("kim", "123",    False),  # 비밀번호 3자 (경계값 미만)
    ("kim", "1234",   True),   # 비밀번호 4자 (경계값)
    ("kim", "kim",    False),  # 아이디 = 비밀번호
])
def test_validate_login_cases(username, password, expected):
    # 5줄의 데이터로 5개의 독립적인 테스트가 생성됨
    assert validate_login(username, password) == expected

06

E2E GUI 자동화 (Playwright)

단위 테스트가 "함수가 올바른가"를 검증한다면, E2E 테스트는 "사용자 입장에서 화면부터 서버까지 전체 흐름이 올바른가"를 검증합니다. Playwright가 실제 브라우저를 켜서 입력, 클릭, URL 이동을 자동으로 수행합니다.

page fixture 이해하기

page는 pytest-playwright가 자동으로 제공하는 fixture입니다. "브라우저 탭 하나"라고 생각하면 됩니다. 테스트 함수 파라미터에 page를 선언하기만 하면 pytest가 자동으로 새 탭을 열어서 주입해주고, 테스트가 끝나면 자동으로 닫힙니다.

이것이 fixture의 핵심입니다. 준비(브라우저 열기)와 정리(닫기)를 자동화해서, 테스트 함수는 핵심 검증 로직에만 집중할 수 있습니다.

tests/e2e/test_login_e2e.py
import pytest

BASE_URL = "http://127.0.0.1:5000"

@pytest.mark.e2e
def test_login_success(page):
    """
    시나리오: 올바른 계정으로 로그인하면 dashboard로 이동하고
    환영 메시지가 표시되어야 한다.
    """
    page.goto(f"{BASE_URL}/login")          # 로그인 페이지 접속
    page.fill("#username", "kim")         # 아이디 자동 입력
    page.fill("#password", "pass123")     # 비밀번호 자동 입력
    page.click("#login-btn")              # 로그인 버튼 클릭

    welcome = page.locator("#welcome")
    assert welcome.inner_text() == "Welcome, kim"


@pytest.mark.e2e
def test_login_fail_shows_error(page):
    """
    시나리오: 빈 아이디로 로그인하면 에러 메시지가 표시되어야 한다.
    """
    page.goto(f"{BASE_URL}/login")
    page.fill("#username", "")             # 빈 아이디로 시도
    page.fill("#password", "pass123")
    page.click("#login-btn")

    error = page.locator("#error")
    assert error.is_visible() is True     # 에러 메시지가 화면에 보여야 함

07

실패 시 스크린샷 자동 저장

E2E 테스트는 실패 원인이 다양합니다. HTML 요소를 못 찾았거나, 페이지 이동이 안 됐거나, 서버가 예상치 못한 응답을 했을 수도 있습니다. 터미널 로그만으로는 "화면이 어떤 상태였는지"를 알기 어렵습니다.

pytest의 hook 기능을 사용해서 테스트 실패 시 그 순간의 화면을 자동으로 캡처하도록 했습니다. hook이란 pytest의 특정 시점에 "끼어들어서" 원하는 코드를 실행하는 기능입니다.

tests/conftest.py
import os
import pytest

@pytest.hookimpl(tryfirst=True, hookwrapper=True)
def pytest_runtest_makereport(item, call):
    """
    pytest hook: 테스트 결과 리포트 생성 시점에 개입
    실패한 E2E 테스트의 스크린샷을 자동 저장
    """
    outcome = yield            # 여기서 실제 테스트 실행
    report = outcome.get_result()

    if report.when == "call" and report.failed:
        page = item.funcargs.get("page", None)  # E2E 테스트만 해당
        if page is not None:
            os.makedirs("artifacts/screenshots", exist_ok=True)
            path = f"artifacts/screenshots/{item.name}.png"
            page.screenshot(path=path, full_page=True)
            print(f"\n📸 스크린샷 저장: {path}")
📸
실무에서의 활용 CI/CD 파이프라인(GitHub Actions 등)에서 테스트가 실패하면 스크린샷이 artifacts로 저장됩니다. 개발자가 직접 환경을 재현하지 않아도 실패 화면을 바로 확인할 수 있어 디버깅 시간이 크게 줄어듭니다. 이 패턴은 실무 QA 팀에서도 표준으로 사용합니다.
